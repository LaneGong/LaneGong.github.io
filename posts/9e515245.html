数据包结构分析

摘要：

本博客主要是记录了利用Wireshark抓每一层协议的数据包并进行分析，不足之处，欢迎指正交流！

本次实验抓包工具：Wireshark

混杂模式

混杂模式就是接收所有经过网卡的数据包，包括不是发给本机的包，即不验证MAC 地址。普通模式下网卡只接收发给本机的包（包括广播包）传递给上层程序，其它的包一律丢弃。一般来说，混杂模式不会影响网卡的正常工作，多在网络监听工具上使用。

关闭和开启混杂模式

  1. 停止抓包

  2. 工具栏➡️Capture➡️options

  3. 选上即开启（默认开启）

    Question: 为什么输入 udp 之后出现那么多种协议呢？

    Answer: DNS和SSDP都是基于udp的传输层之上的协议

    扩展：客户端向 DNS 服务器查询域名，一般返回的内容都不超过 512 字节，用 UDP 传输即可。不用经过三次握手，这样 DNS 服务器负载更低，响应更快。理论上说，客户端也可以指定向 DNS 服务器查询时用 TCP，但事实上，很多 DNS 服务器进行配置的时候，仅支持 UDP 查询包。

ARP协议

  • Wireshark filter:arp

  • 抓到的ARP包

  • 分析NO.5646请求包

    通过抓包列表显示这个数据包是中国移动服务器广播发出的ARP请求包，主要内容是Who has 192.168.1.3?谁有该IP地址（本机IP），请求该IP的MAC地址。Tell 192.168.1.1，请将MAC地址告知192.168.1.1（请求方），发送ARP响应包。

    查看Address Resolution Protocol(request) ARP请求包内容

    Address Resolution Protocol (request)#ARP地址解析协议，request表示请求包
    Hardware type: Ethernet (1)#硬件类型
    Protocol type: IPv4 (0x0800)#协议类型
    Hardware size: 6#硬件地址
    Protocol size: 4#协议长度
    Opcode: request (1)#操作码，值为1表示ARP请求包
    Sender MAC address: ChinaMob_38:57:c1 (90:47:3c:38:57:c1)#源的MAC地址
    Sender IP address: 192.168.1.1#源的IP地址
    Target MAC address: 00:00:00_00:00:00 (00:00:00:00:00:00)#目标MAC地址，广播
    Target IP address: 192.168.1.3#目标IP地址

  • 分析NO.5647应答包

    首先通过抓包列表可以清楚发现应答包已经不是广播包了，单播通信，是从目标IP发送到请求IP的主机上，主要信息内容为目标地址IP所对应的MAC地址。

    查看Address Resolution Protocol(reply) ARP应答包内容

    Address Resolution Protocol (reply)#ARP地址解析协议，reply表示应答包
    Hardware type: Ethernet (1)#硬件类型
    Protocol type: IPv4 (0x0800)#协议类型
    Hardware size: 6#硬件地址
    Protocol size: 4#协议长度
    Opcode: reply (2)#操作码，值2表示ARP应答包
    Sender MAC address: Apple_01:30:d7 (3c:06:30:01:30:d7)#源MAC地址
    Sender IP address: 192.168.1.3#源IP地址
    Target MAC address: ChinaMob_38:57:c1 (90:47:3c:38:57:c1)#目标MAC地址
    Target IP address: 192.168.1.1#目标IP地址

  • 总结：应答包补全了自己的MAC地址，目的地址和源地址做了替换

  • 分析TCP断开连接，四次挥手过程

    重抓了一下，帧数可能有些不一致，但无大影响

    命令： curl -I baidu.com

    curl 是一个在命令行下工作的文件传输工具，我们这里用来发送 http 请求。-I 表示仅返回头部信息。

    第一次挥手：主机发送一个[FIN+ACK]，表示自己没有数据要发送了，想断开连接，并进入FIN_WAIT_1 状态

    第二次挥手：百度收到 FIN 后，知道不会再有数据从主机传来，发送 ACK 进行确认，确认序号为收到序号+1（与SYN相同，一个FIN占用一个序号），客户端进入 CLOSE_WAIT 状态。

    第三次挥手：百度发送 [FIN+ACK] 给对方，表示自己没有数据要发送了，百度进入LAST_ACK 状态，然后直接断开 TCP 会话的连接，释放相应的资源。

    第四次挥手：主机收到了百度的FIN命令后，进入 TIMED_WAIT 状态，并发送 ACK 确认消息。主机在 TIMED_WAIT 状态下，等待一段时间，没有数据到来，就认为对面已经收到了自己发送的ACK 并正确关闭了进入 CLOSE 状态，自己也断开了 TCP 连接，释放所有资源。当百度收到主机的ACK 回应后，会进入 CLOSE 状态并关闭本端的会话接口，释放相应资源。

  • 四次挥手的小插曲 Spurious Retransmission 和 out of order

    使用curl www.baidu.com抓包如下

    TCP Retransmission

    在网络环境中，数据包丢失是非常常见的事情。TCP协议有内建的处理机制来确保网络数据传输正确。但是，如果看到大量的TCP重传（TCP Retransmission），则往往表示存在异常。

    当数据发送方没有接收到接收方的ACK，就会TCP Retransmission。虽然很多时候是由于网络链路问题，但是也有可能是服务器压力过大或者服务器端网卡、驱动等主机故障导致。

    TCP Retransmission有三种类型，重点展开遇到的 Spurious Retransmission ：

    Checks for a retransmission based on analysis data in the reverse direction. Set when all of the following are true:

    • The SYN or FIN flag is set.
    • This is not a keepalive packet.
    • The segment length is greater than zero.
    • Data for this flow has been acknowledged. That is, the last-seen acknowledgement number has been set.
    • The next sequence number is less than or equal to the last-seen acknowledgement number.

    满足了上述条件，设置虚假重传。

    TCP Out-Of-Order

    Set when all of the following are true:

    • This is not a keepalive packet.
    • In the forward direction, the segment length is greater than zero or the SYN or FIN is set.
    • The next expected sequence number is greater than the current sequence number.
    • The next expected sequence number and the next sequence number differ.
    • The last segment arrived within the Out-Of-Order RTT threshold. The threshold is either the value shown in the “iRTT” (tcp.analysis.initial_rtt) field under “SEQ/ACK analysis” if it is present, or the default value of 3ms if it is not.

    Supersedes “Retransmission”.

    reference：TCP Analysis

HTTP协议

  • Terminal命令： curl -I baidu.com ，抓包

    筛选 TCP 协议因为 HTTP 是 TCP 的上层协议，所以我们过滤 TCP 的数据会包含 HTTP 协议的数据包，同时也能观察到前面的TCP三次握手、四次挥手。第158和160是HTTP数据包。

  • 第一步：发送了一个 HTTP 的 HEAD 请求

  • 第二步：服务器收到我们的请求返回了一个 Seq/ACK 进行确认

  • 第三步：服务器将 HTTP 的头部信息返回给我们客户端，状态码为 200 表示页面正常

  • 第四步：客户端收到服务器返回的头部信息向服务器发送 Seq/ACK 进行确认

  • 最后：发送完成之后客户端就会发送 FIN/ACK 来进行关闭链接的请求。
